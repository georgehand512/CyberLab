# exp_003_Re-run_Enumeration_With_Logging

## Experiment Title

Re-run Internal Network Enumeration with Enhanced Windows Logging Enabled

---

# 1. Objective

The objective of this experiment is to repeat the internal enumeration activity previously conducted in exp_001_Local_Enumeration while enhanced Windows audit logging is enabled on the target system, as enabled during exp_002_Enable_Enhanced_Logging.

The purpose is to directly compare host telemetry generated during reconnaissance activity under two conditions:

• Default Windows logging configuration (exp_001_Local_Enumeration)  
• Enhanced audit logging configuration (exp_002_Enable_Enhanced_Logging)

This experiment aims to:

- Observe the telemetry generated when internal reconnaissance occurs
- Compare logging visibility against the default Windows configuration
- Identify specific Windows security events triggered by enumeration activity
- Understand how enhanced audit policies improve detection capability
- Evaluate how reconnaissance behaviour appears from a SOC monitoring perspective

Unlike exp_001_Local_Enumeration, the primary focus of this experiment is not service discovery results, but the telemetry generated by the Windows host when it receives repeated inbound connection attempts.

# 4. Methodology

### Scan Timing Consideration

During exp_001_Local_Enumeration it was observed that the Windows Defender Firewall blocks unsolicited inbound TCP connections, causing Nmap to classify most scanned ports as "filtered".

When ports are filtered rather than explicitly closed, Nmap assumes packets may have been lost and performs multiple retransmission attempts before determining the port state. This behaviour can significantly increase scan duration, particularly during large or full port scans.

To prevent excessively long scan times while still generating sufficient inbound connection attempts for telemetry analysis, certain Nmap timing and retry parameters may be adjusted in this experiment.

- Event ID 5156 – Windows Filtering Platform allowed connection
- Event ID 5157 – Connection blocked by filtering platform
- Event ID 4624 – Successful logon (if applicable)
- Event ID 4625 – Failed logon (if triggered)


SOC Analysis Consideration:
When reviewing logs, analysts should not only examine individual events but also look for behavioural patterns that may indicate automated scanning activity.

Indicators of possible reconnaissance include:

- Multiple blocked connection attempts originating from the same source IP
- Sequential connection attempts across many destination ports
- Rapid bursts of Windows Filtering Platform events occurring within a short timeframe

# 7. Detection & Mitigation Reflection

Comparison with exp_001_Local_Enumeration:

exp_001_Local_Enumeration demonstrated that internal enumeration activity generated minimal observable telemetry under the default Windows logging configuration. Although connection attempts were blocked by the Windows Defender Firewall, the operating system produced little actionable logging.

In this experiment, enhanced audit policies and firewall logging were enabled prior to repeating the same reconnaissance activity. The goal is to evaluate whether this configuration change improves detection visibility.

Key comparison areas include:

- Visibility of inbound connection attempts
- Windows Filtering Platform events generated during scans
- Firewall log entries showing allowed or dropped traffic
- The ability to identify patterns of repeated connection attempts from the same source

Key considerations:

- Increased visibility of inbound connection attempts
- Correlation opportunities across multiple event types
- Detection potential for abnormal scanning behaviour

Potential detection mechanisms:

- SIEM correlation of repeated filtering platform events
- IDS/IPS network detection
- Endpoint telemetry analysis

Mitigation considerations:

- Alert on excessive connection attempts
- Monitor rapid sequential port access
- Apply host-based firewall rules where appropriate

---

# 8. Key Takeaway

Re-running the enumeration activity with enhanced Windows logging enabled demonstrates how increased telemetry significantly improves the visibility of reconnaissance behaviour.

While the scanning activity itself remains unchanged, the operating system now generates additional security events that provide valuable context for security monitoring.

Events generated by the Windows Filtering Platform offer insight into inbound connection attempts, allowing security teams to identify patterns consistent with port scanning or network probing.<br>
Service detection may also elicit responses from previously hidden services (e.g., port 5357), illustrating the importance of deeper probes for full telemetry analysis.

This experiment highlights the importance of proper audit configuration in enabling effective detection. Without adequate logging, early-stage attacker activity may occur with minimal visibility.

# 9. Lessons Learned

Technical:

- Nmap scanning generates multiple inbound connection events on the Windows host
- Windows Filtering Platform events provide detailed network connection telemetry
- Service detection scans generate more logging activity than simple host discovery
- Scanning hosts protected by a firewall that silently drops packets can significantly increase Nmap scan duration due to retransmission behaviour.<br>
  Adjusting retry and timeout parameters can help control scan time while still generating meaningful telemetry for analysis.
- Some services may not respond reliably to SYN-based scanning and can appear as filtered. <br>Service version detection (-sV) performs deeper probing, which can elicit responses from these services and result in ports being identified as open. <br>This behaviour was observed with port 5357, which was only identified during service detection.
- Full port scans across all 65,535 TCP ports can result in significant execution time when targeting hosts protected by firewalls that silently drop packets. <br>In this experiment, the scan did not complete due to host timeout, but still generated substantial logging, demonstrating that full scan completion is not required for effective detection.

Detection:

- Enhanced logging dramatically increases visibility of reconnaissance behaviour
- Correlation of repeated connection events can indicate scanning activity
- Host-based logging is a valuable supplement to network monitoring

Operational:

- Logging configuration changes from exp_002_Enable_Enhanced_Logging remained stable
- Evidence collection procedures validated
- Structured experiment comparison possible across experiments
